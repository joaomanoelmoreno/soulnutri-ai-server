## 6. Gargalo Principal (root cause)

🔴 **GARGALO: ONNX Inference do CLIP rodando em CPU do Render Standard 1GB**

**Evidências:**
1. Backend interno responde por **75% do tempo total** (1568ms de 2103ms).
2. Dentro do backend, a inferência ONNX do CLIP é, historicamente, a fase dominante (logs `[TIMING] Embedding (ONNX/PyTorch): ~1000-1800ms` no histórico do projeto).
3. A similaridade numpy (~5-50ms) e cache hit (~10-50ms) são triviais.
4. Tempo escala parcialmente com tamanho da imagem (Atum 3.5MB → 2291ms backend; Abóbora 128KB → 1133ms backend), o que indica que parte do custo está na decodificação + redimensionamento da imagem **antes** da inferência ONNX.

**Gargalos secundários:**
- 🟡 **Upload de imagens grandes** (3.5MB → ~838ms só de transferência). Câmeras de celular modernas geram fotos de 2-5MB. Sem compressão client-side, isso adiciona 1-2s no 4G.
- 🟢 **Render CPU compartilhada (Standard 1GB)**: hardware genérico, sem aceleração SIMD/AVX otimizada para ONNX.

## 8. Recomendações — APENAS DIAGNÓSTICO, NÃO EXECUTAR AGORA

Lista priorizada, sem implementação:

### 🥇 Maior impacto potencial (cortaria 800-1300ms)

#### R1. Mover inferência CLIP para GPU
- **Onde:** Render → migrar para tier com GPU (Render Pro GPU, Modal, Replicate ou Fly.io GPU)
- **Ganho estimado:** ONNX inference cai de ~1500ms (CPU) para ~50-150ms (GPU NVIDIA T4)
- **Custo:** GPU = US$0,50–US$1,50/hora vs CPU US$0,10/hora. Vale a pena se SLA <500ms é diferencial competitivo.
- **Risco:** baixo — mesma API, mesmo código, só infra.

#### R2. CLIP no navegador (WebAssembly / WebGPU)
- **Onde:** Frontend Vercel — rodar CLIP ONNX direto no celular do usuário
- **Ganho estimado:** elimina rede + upload completamente. Tempo cai para 200-600ms (depende do dispositivo).
- **Custo:** zero infra adicional, mas modelo precisa caber no bundle (ViT-B-16 = ~150MB; usar versão quantizada INT8 = ~50MB).
- **Risco:** médio — celulares antigos podem ficar lentos. Bundle PWA cresce.

### 🥈 Impacto moderado (cortaria 300-500ms)

#### R3. Compressão de imagem client-side antes do upload
- **Onde:** Frontend, antes do POST
- **Como:** redimensionar para 512×512px e comprimir para JPEG quality 85 (~50-150KB) antes de enviar
- **Ganho estimado:** elimina ~500-800ms de upload em 4G. Reduz também o tempo de decodificação no backend.
- **Custo:** zero. Risco: baixo (CLIP processa em 224×224 mesmo, então qualidade não importa).

#### R4. Otimizar ONNX runtime no CPU atual
- **Como:** ativar SIMD/AVX2 explicitamente, threadpool tuning, INT8 quantization do modelo
- **Ganho estimado:** 30-40% no tempo de inferência (1500ms → ~900-1000ms)
- **Custo:** ~1 dia de tuning. Risco: regressão de acurácia se quantização for agressiva.

### 🥉 Impacto pequeno mas barato (cortaria 50-150ms)

#### R5. Render warm-keepalive
- **Onde:** Render Standard tem cold-start ocasional após inatividade
- **Como:** cron job pingando `/api/health` a cada 5 min
- **Ganho:** elimina cold-start de ~5-15s no primeiro request após idle
- **Custo:** zero. Risco: zero.

#### R6. HTTP/2 push de pré-warming
- Não cabe no caso atual — ignorar.

#### R7. Cache mais agressivo
- Já existe (cache_service). Não há ganho adicional sem mudar política.

Para preencher os cenários "celular Wi-Fi Cibi Sana" e "celular 4G/5G", siga:

### Opção A — Chrome DevTools Remote Debug (RECOMENDADO)

1. Conecte o celular ao Mac via USB
2. Abra Chrome no Mac → `chrome://inspect/#devices`
3. Habilite "USB debugging" no celular (Configurações → Opções de Desenvolvedor)
4. No celular, abra `https://soulnutri.app.br` no Chrome
5. No Mac aparece o dispositivo → clique "Inspect"
6. Aba **Network** → ativar "Preserve log" e "Disable cache"
7. Filtrar por `identify`
8. Faça 5 scans no celular
9. Para cada request `/api/ai/identify`, anote da aba Network:
   - **Size** (tamanho do upload + download)

   - **Waterfall → Content Download**

4. Pare **no instante em que o resultado aparece**

## 10. Resumo Executivo (1 minuto)

- **Meta:** scan <500ms.
- **Realidade:** 1530-3184ms (média 2103ms). **Meta NÃO atingida em 5/5 testes.**
